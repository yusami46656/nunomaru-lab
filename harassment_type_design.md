## 0. プロジェクト概要

### 目的
16Personalities風のUIで、自分のハラスメント傾向を診断するエンタメ性の高いWeb診断ツールを提供する。

### 配置先
- 親サイト: `https://www.nunomaru-lab.com/`(Next.js)
- 配置パス: `/experiments/harassment-type/` 配下

### URL構造
| パス | 役割 |
|---|---|
| `/experiments/harassment-type/` | トップ・診断スタート画面 |
| `/experiments/harassment-type/quiz/` | 質問回答画面(SPA的に1画面で進行) |
| `/experiments/harassment-type/result/[typeId]/` | 結果ページ。共有用URL |

`typeId` は後述の16タイプの英数字キー(`emperor`, `villager`, `strategist` など)。

### 技術スタック
- フレームワーク: **Next.js**(親サイトに準拠)
- スタイリング: 親サイトの方針に合わせる(Tailwind CSS推奨)
- データ集計: **Google Apps Script (GAS) Webhook + Google スプレッドシート**(完全無料)
- 画像素材: 後からPNG素材を `/public/experiments/harassment-type/images/` 等に配置予定

### デザイン方針
- 16Personalities風のトンマナ(親サイトのトンマナとはあえて揃えない)
- 各タイプにイメージカラー(Main / Pale 2色)とイラスト枠を用意
- スマホファースト、白基調・カラフルなアクセント

### 4属性の定義

| 属性 | 名称 | 傾向 | キーフレーズ |
|---|---|---|---|
| **P** | パワハラ | 自分の地位・権限・優位性を使い、相手を従わせようとする傾向 | **上から支配する** |
| **M** | モラハラ | 自分の正しさ・常識・価値観を使い、相手を否定・支配しようとする傾向 | **内面を裁く** |
| **S** | セクハラ | 相手の身体・性・恋愛・私生活・親密さの境界を軽視して踏み込む傾向 | **境界を越える** |
| **C** | カスハラ | 相手の役割・責任・対応義務につけ込み、過剰な要求や不満をぶつける傾向 | **役割につけ込む** |

## 1. 全体フロー

```
[トップ画面]
    ↓ 「診断を始める」クリック
[質問画面] ← Q1〜Q16を順番に表示
    ↓ Q16回答後、自動で集計
    ↓ POST /api/submit (集計データ送信)
    ↓
[結果画面] (/result/[typeId]/)
    ↓
    ├─ シェアボタン(X / Instagram / LINE / Facebook)
    ├─ URLコピーボタン
    └─ もう一度診断する
```

## 2. 画面仕様

### 2-1. トップ画面 `/experiments/harassment-type/`

**目的**: 診断の概要説明と開始導線

**コンテンツ要素**:
- ヒーロー: タイトル「ハラスメントタイプ診断」、サブタイトル「あなたの中に潜むモンスター、診断します」(仮)
- 診断概要: 「全16問・約5分で完了」「16タイプから判定」
- **4属性の説明セクション**(後述「2-1-1. 4属性の説明セクション」)
- **結果の見方セクション**(後述「2-1-2. 結果の見方セクション」)
- 16タイプの一覧プレビュー(イラスト+名前のグリッド、タップで各タイプの説明モーダルでも可)
- 「診断を始める」ボタン → `/quiz/` へ遷移
- 注意書き: 「本診断はエンタメ目的です。実際のハラスメントの判定基準を示すものではありません」

#### 2-1-1. 4属性の説明セクション

診断で測る4つの属性(P / M / S / C)を、ユーザーが診断前にざっくり把握できるよう簡潔に提示する。

**表示内容**: 4属性をカード or アイコン+短文の形式で並べる。各カードには以下を含める。
- 属性記号(P / M / S / C)
- 属性名(パワハラ / モラハラ / セクハラ / カスハラ)
- キーフレーズ(「上から支配する」「内面を裁く」「境界を越える」「役割につけ込む」)
- 1行の補足説明

**文言案**:

| 属性 | 名称 | キーフレーズ | 補足 |
|---|---|---|---|
| P | パワハラ | 上から支配する | 立場や権限を使って従わせようとする傾向 |
| M | モラハラ | 内面を裁く | 正しさや常識を盾に相手を否定する傾向 |
| S | セクハラ | 境界を越える | 身体・恋愛・私生活へ無遠慮に踏み込む傾向 |
| C | カスハラ | 役割につけ込む | 役割や責任を盾に過剰な要求をする傾向 |

**配置**: 「診断を始める」ボタンの上に置く。スマホでは縦積み、PCでは横並び4カラム。

#### 2-1-2. 結果の見方セクション

ユーザーがありがちな誤解(「属性が多い=ヤバい」「皇帝がラスボスっぽくて格好いい」等)を防ぎ、診断の意図を正しく伝える短い説明。

**伝えたいこと(原則)**:
1. **属性は「多い/少ない」ではなく、「%が高いか」を見る** ─ 該当属性が多くても各%が低ければ問題は小さい。逆に1属性だけでも%が高ければ要注意。
2. **%が高いほど、その傾向が日常で出やすい** ─ %は「どれくらい強くその傾向を持っているか」の目安。
3. **診断はエンタメであり、ラベル貼りではない** ─ 自分や他人を断罪するためではなく、自分のクセに気づく入り口として使ってほしい。

**文言案**(本文として表示):

> **結果の見方**
>
> この診断では、4つの属性(P / M / S / C)それぞれについて、あなたがどれくらい強くその傾向を持っているかを%で表示します。
>
> **大事なのは「%の高さ」です。** 該当する属性の数が多いことが悪いわけではありません。たとえ1つの属性でも、%が高ければその傾向は日常で出やすくなります。逆に複数の属性に該当していても、それぞれの%が低ければ、注意度はそれほど高くありません。
>
> あくまでエンタメ診断ですが、自分のクセに気づくきっかけとして楽しんでください。

**配置**: 4属性の説明セクションのすぐ下、もしくは16タイプ一覧プレビューの直前。アコーディオンで折りたためる形式でもよい(スマホで縦に長くなりすぎないため)。

**SEO**:
- title: `ハラスメントタイプ診断 | nunomaru-lab`
- description: 16タイプから判定する診断であることを記載
- OGP画像: 診断トップ用の画像(後からPNG差し込み)

### 2-2. 質問画面 `/experiments/harassment-type/quiz/`

**目的**: Q1〜Q16の出題と回答

**コンテンツ要素**:
- プログレスバー(現在 X / 16)
- 質問文
- 4択の選択肢(A〜D)
- 「戻る」ボタン(Q2以降で表示。ただし戻った時の選択状態は保持)
- 選択肢を選んだら自動で次の質問へ遷移(タップ数最小化)

**振る舞い**:
- 状態は React state で管理(SSR不要、Client Componentで実装)
- リロードで状態は失われる(中断保存はしない)
- 全16問答え終わったら、判定処理を実行 → 集計データをPOST → 結果画面へリダイレクト

**選択肢シャッフル**:
- A〜Dの表示順は質問ごとにランダム化することを推奨(「いつも上から選ぶ人」のバイアス除去)
- ただし、配点は元の選択肢に紐づく(表示順とは独立)

### 2-3. 結果画面 `/experiments/harassment-type/result/[typeId]/`

**目的**: 診断結果の表示と共有

**コンテンツ要素**:
- 該当タイプのイラスト(PNG埋込予定)
- タイプ名(例: 「あなたは ハラスメント皇帝」)
- タイプのキャプション(説明文)
- 口癖(3つ)
- 能力(2つ:能力名と説明)
- タイプへのアドバイス(2つ)
- 各属性スコア(P / M / S / C)を %表示+バーで可視化
- アクションボタン:
  - **シェアボタン群**(X / Instagram / LINE / Facebook の4種、各SNSのアイコンを横並び表示) → 後述のシェア仕様参照
  - 「URLをコピー」 → 現在のページURLをクリップボードに
  - 「もう一度診断する」 → トップへ戻る
- 「他のタイプを見る」 → 16タイプ全部を見られるリンク or モーダル

**重要な仕様**:
- 結果URLは `[typeId]` ベースなので、シェアされた人が同じタイプの結果ページを見ることになる
- ただし、「実際の自分のスコア」は本人だけが見えるよう、URL に直接スコアを埋め込まない
- 自分のスコアをSNSで共有したい場合は、シェア用テキストにスコアサマリを含める(X / Facebook / LINE)。Instagramはテキストシェアに非対応のため、画像保存+リンクコピー方式となる

**SEO/OGP**:
- 各 `[typeId]` ごとに動的に title / description / OGP画像 を切り替える
- `generateMetadata` で実装
- OGP画像は各タイプ用のPNG(後から差し込み)

### 2-4. エラー画面・存在しないタイプ
- 不正な `typeId` の場合は 404 へ
- API エラー時(集計POSTが失敗)はユーザーには見せず、結果表示は通常通り行う

## 3. データモデル

### 3-1. 質問データ

各質問は以下の構造で `data/questions.ts` (もしくは JSON) に保持する。

```typescript
type Choice = {
  id: 'A' | 'B' | 'C' | 'D';
  text: string;
  scores: {
    P: number;  // パワハラ配点
    M: number;  // モラハラ配点
    S: number;  // セクハラ配点
    C: number;  // カスハラ配点
  };
};

type Question = {
  id: string;        // "Q1" 〜 "Q16"
  number: number;    // 1〜16
  scenario: string;  // 質問文
  choices: Choice[];
};
```

質問内容と配点は **本書 Appendix A** に全16問掲載。

### 3-2. タイプデータ

各タイプは以下の構造で `data/types.ts` に保持する。

```typescript
type Ability = {
  name: string;         // 能力名(例: "会議室チェックメイト")
  description: string;  // 能力の説明
};

type HarassmentType = {
  id: string;             // "villager", "strategist", "emperor" など
  number: number;         // 1〜16
  name: string;           // "ハラスメント皇帝"
  attributes: ('P'|'M'|'S'|'C')[];  // 該当属性
  attributesLabel: string;          // "PMSC", "PM", "" など表示用ラベル
  caption: string;        // タイプのキャプション(説明文)
  phrases: string[];      // 口癖(3つ)
  abilities: Ability[];   // 能力(2つ)
  advice: string[];       // アドバイス(2つ)
  colors: {
    main: string;         // テーマカラー Main(例: "#2B2A2E")
    pale: string;         // テーマカラー Pale(例: "#F1F0F2")
  };
  imagePath: string;      // PNGパス。後から差し込み(初期はプレースホルダー)
};
```

タイプの英数字キー対応表とフルコンテンツは **本書 Appendix B** 参照。

### 3-3. 判定結果(クライアント側で算出、サーバには送信)

```typescript
type DiagnosisResult = {
  scores: {
    P: number;  // 0-100の整数(%)
    M: number;
    S: number;
    C: number;
  };
  candidates: ('P'|'M'|'S'|'C')[];  // 40%以上の属性
  typeId: string;                    // 確定タイプのID
  answers: ('A'|'B'|'C'|'D')[];      // ユーザーが選んだ選択肢の配列(Q1〜Q16の順)
};
```

### 3-4. 集計用送信データ(サーバへPOST)

```typescript
type SubmissionPayload = {
  timestamp: string;        // ISO8601。クライアントタイムスタンプ
  typeId: string;
  scores: { P: number; M: number; S: number; C: number };
  answers: string;          // "BCDABACDABCDABCD" のような16文字の文字列(Q1からQ16の選択肢)
};
```

**個人情報は一切送信しない**。IPアドレス・User-Agent・Cookie・端末識別子なども保存しない。

## 4. 判定ロジック

### 4-1. 配点とスコア算出

各属性の最大獲得点は質問データから動的に算出する(ハードコードしない)。

```typescript
// 各属性の最大点を計算
function calcMaxScore(questions, attr) {
  return questions.reduce((sum, q) =>
    sum + Math.max(...q.choices.map(c => c.scores[attr])), 0);
}
// MaxP=50, MaxM=66, MaxS=56, MaxC=43 (Appendix A の配点に基づく)
```

ユーザーの回答からスコアを算出:

```typescript
function calcScore(answers, questions, attr) {
  const max = calcMaxScore(questions, attr);
  const raw = answers.reduce((sum, choiceId, i) => {
    const choice = questions[i].choices.find(c => c.id === choiceId);
    return sum + choice.scores[attr];
  }, 0);
  return Math.round(Math.max(0, raw) / max * 100);  // 0-100の整数
}
```

### 4-2. タイプ判定アルゴリズム

```
Step 1. P/M/S/C それぞれのスコアを算出 (0-100%)
Step 2. 40%以上の属性を「候補」とする
Step 3. 候補数で分岐:
  - 候補0個 → "villager"(無害な村人)
  - 候補1個 → その属性の単独タイプ
  - 候補2個以上:
      - Top - Second >= 20% なら Top単独タイプ
      - Top - Second < 20% なら 候補すべてを保有する複合タイプ
```

候補属性のセットからタイプIDを引く対応表は Appendix B を参照。

### 4-3. 実装上の注意

- スコアは整数(%)で扱い、四捨五入は `Math.round()` で行う
- 減点で属性合計がマイナスになった場合は 0% として扱う(`Math.max(0, raw)`)
- 候補属性のソート順は固定(P→M→S→C の順で内部処理)
- タイプIDの参照は **属性のセット**で行う(順序非依存)

## 5. API設計

### 5-1. POST `/api/submit`

**目的**: 集計データを Google スプレッドシートに送信

**リクエスト**:
```json
{
  "timestamp": "2026-05-08T12:34:56.789Z",
  "typeId": "emperor",
  "scores": { "P": 75, "M": 80, "S": 65, "C": 70 },
  "answers": "BCDABACDABCDABCD"
}
```

**レスポンス**:
- 成功: 200 OK `{ "ok": true }`
- 失敗: 任意のステータス(クライアントは無視して結果表示を続行)

**実装方針**:
- Next.js の Route Handler (`app/api/submit/route.ts`) で実装
- 環境変数 `GAS_WEBHOOK_URL` に GAS の Webhook URL を保持
- リクエスト本文をそのまま GAS にプロキシして送る
- タイムアウトは 5秒程度
- エラーが発生してもユーザー体験には影響させない(必ず200を返すか、失敗してもクライアントが無視する設計)

### 5-2. Google Apps Script (GAS) 側の実装イメージ

```javascript
function doPost(e) {
  const sheet = SpreadsheetApp.openById('SPREADSHEET_ID')
                              .getSheetByName('responses');
  const data = JSON.parse(e.postData.contents);
  sheet.appendRow([
    data.timestamp,
    data.typeId,
    data.scores.P,
    data.scores.M,
    data.scores.S,
    data.scores.C,
    data.answers
  ]);
  return ContentService.createTextOutput(JSON.stringify({ ok: true }))
    .setMimeType(ContentService.MimeType.JSON);
}
```

**スプレッドシートのカラム**:
| timestamp | typeId | P | M | S | C | answers |
|---|---|---|---|---|---|---|

### 5-3. セキュリティ考慮
- GAS の Webhook URL は環境変数で管理(クライアントには露出しない)
- レート制限は GAS側 or Vercel側で必要に応じて
- スパム対策として、サーバ側で簡易バリデーション(typeId が16タイプのいずれか、スコアが0-100の整数、answers が16文字)

## 6. シェア機能

### 6-0. 全体方針

結果画面のシェアボタンは X / Instagram / LINE / Facebook の4種に対応する。各SNSのアイコンを横並びで表示し、タップで各SNSのシェアフローを起動する。

**共通のシェアテキスト**(typeName と scores はランタイムに動的生成):

```
私のハラスメントタイプは「{typeName}」でした

P:{P}% / M:{M}% / S:{S}% / C:{C}%

#ハラスメントタイプ診断
{resultUrl}
```

例:
```
私のハラスメントタイプは「ハラスメント皇帝」でした

P:75% / M:80% / S:65% / C:70%

#ハラスメントタイプ診断
https://www.nunomaru-lab.com/experiments/harassment-type/result/emperor/
```

各SNSの仕様差は以下のとおり。

| SNS | 方式 | テキスト | URL | 画像 | 備考 |
|---|---|---|---|---|---|
| X | Web Intent | ◯ | ◯ | OGP経由 | 標準的 |
| Facebook | Sharer | △ (quote) | ◯ | OGP経由 | テキストはOGP優先表示 |
| LINE | Share URL | ◯ | ◯ | OGP経由 | スマホ実機で挙動が安定 |
| Instagram | 画像保存+クリップボード | ✕ | クリップボード経由 | アプリへ画像送出 | テキストシェアAPIなし |

### 6-1. X シェア

**仕様**:
- 「Xでシェア」アイコン押下で、X(旧Twitter)の Web Intent を新規タブで開く
- シェアテキストには共通フォーマット(タイプ名+スコア+ハッシュタグ+結果URL)を含める

**実装**:
```typescript
const text = buildShareText(typeName, scores, resultUrl);
const shareUrl = `https://twitter.com/intent/tweet?text=${encodeURIComponent(text)}`;
window.open(shareUrl, '_blank', 'noopener,noreferrer');
```

### 6-2. Facebook シェア

**仕様**:
- 「Facebookでシェア」アイコン押下で、Facebook Sharer を新規タブで開く
- Facebook は OGP の og:title / og:description / og:image を優先表示するため、結果ページの OGP を `[typeId]` ごとに正しく設定しておくことが重要(6-6 参照)
- `quote` パラメータでスコア込みのテキストを補足する

**実装**:
```typescript
const text = buildShareText(typeName, scores, resultUrl);
const shareUrl =
  `https://www.facebook.com/sharer/sharer.php` +
  `?u=${encodeURIComponent(resultUrl)}` +
  `&quote=${encodeURIComponent(text)}`;
window.open(shareUrl, '_blank', 'noopener,noreferrer');
```

### 6-3. LINE シェア

**仕様**:
- 「LINEでシェア」アイコン押下で、LINE の Share URL を新規タブで開く
- スマホ実機ではLINEアプリが起動し、PCではLINE Web版が開く
- テキスト+結果URLをそのまま送出する

**実装**:
```typescript
const text = buildShareText(typeName, scores, resultUrl);
const shareUrl = `https://line.me/R/share?text=${encodeURIComponent(text)}`;
window.open(shareUrl, '_blank', 'noopener,noreferrer');
```

### 6-4. Instagram シェア

**仕様(注意)**:
Instagram は外部サイトからのテキストシェアAPIを公開していない。Instagramフィード/ストーリーへの直接ポストは不可なため、**「シェア用画像のダウンロード+結果URLのクリップボードコピー」** を組み合わせて実現する。

**フロー**:
1. ボタン押下で、結果画面のサマリ画像(タイプ名+スコア+ロゴ等を含むPNG)を生成 or 事前用意のタイプ別シェア画像を取得
2. 画像をユーザー端末にダウンロード(または共有シート経由でアプリ送出)
3. 結果URL + シェアテキストを `navigator.clipboard.writeText()` でクリップボードにコピー
4. トーストで「画像を保存し、テキストをコピーしました。Instagramにペーストしてシェアしてください」と案内

**実装方針**:
- スマホで `navigator.share()` (Web Share API) が使える場合はそちらを優先(共有シートからInstagramを選んでもらう方が体験が良い)
- 使えない環境では上記の「画像DL+テキストコピー」方式にフォールバック

```typescript
async function shareToInstagram(typeId: string, typeName: string, scores: Scores, resultUrl: string) {
  const text = buildShareText(typeName, scores, resultUrl);
  const imagePath = `/experiments/harassment-type/share/${typeId}.png`;

  // Web Share API が画像送信に対応していれば優先
  if (navigator.canShare?.({ files: [] })) {
    try {
      const blob = await fetch(imagePath).then(r => r.blob());
      const file = new File([blob], `${typeId}.png`, { type: 'image/png' });
      if (navigator.canShare({ files: [file] })) {
        await navigator.share({ files: [file], text });
        return;
      }
    } catch (_) { /* フォールバックへ */ }
  }

  // フォールバック: 画像DL + テキストコピー
  const a = document.createElement('a');
  a.href = imagePath;
  a.download = `${typeId}.png`;
  a.click();
  await navigator.clipboard.writeText(text);
  showToast('画像を保存し、テキストをコピーしました。Instagramにペーストしてシェアしてください');
}
```

**画像素材(A案: 静的16枚方式を採用)**:

Instagramへ送出する画像は、**OGP画像とは別物として、タイプごとに静的PNGを16枚用意する**(`villager`〜`emperor`)。OGP画像の流用は避ける。

- 配置パス: `/public/experiments/harassment-type/share/{typeId}.png`
- サイズ: **1080×1080 (正方形)** ─ Instagramフィードの推奨サイズ
- 含めるべき要素:
  - タイプ名(例: 「ハラスメント皇帝」)
  - タイプのキャラクター/イラスト
  - テーマカラー(Main / Pale)
  - 診断のロゴ・タイトル「ハラスメントタイプ診断」
  - 診断URL or QRコード(Instagramは投稿本文にリンクを貼れないため、画像内に誘導を入れる)
- **含めないもの**: ユーザー個人のスコア(P:75% など)。スコアはユーザーごとに変動するため静的画像では対応不可。スコア共有はX / LINE / Facebook のテキストシェアで担保する。

**OGP画像と分けて用意する理由**:

| 用途 | 画像パス | サイズ | 役割 |
|---|---|---|---|
| OGP(X / Facebook / LINE) | `og/{typeId}.png` | 1200×630 (横長) | リンクのサムネイル(副次的) |
| Instagram | `share/{typeId}.png` | 1080×1080 (正方形) | 投稿の主役。画像単体で診断内容が伝わる必要がある |

OGP画像は横長で「リンクの説明補助」、Instagram画像は正方形で「画像が投稿の主役」と役割が違うため、流用はせず別途作成する。

**将来拡張の余地**:
- 個人のスコアを画像に焼き込みたい場合は、`next/og` などで動的画像生成へ拡張可能(B案)。本Phase時点ではA案で進め、需要が出たら検討する。

### 6-5. URLコピー

- 現在のページURL(結果ページのURL)を `navigator.clipboard.writeText()` でコピー
- コピー成功時はトースト表示「URLをコピーしました」

### 6-6. OGP対応

X / Facebook / LINE のシェアでは、シェア先での表示は結果ページのOGPに依存する。

- 各タイプの結果ページ `[typeId]` ごとに動的にOGP画像を切り替え
- `generateMetadata` で `openGraph.images` および `twitter.card` にタイプ別画像を指定
- 画像パス: `/public/experiments/harassment-type/og/{typeId}.png`(後から差し込み)
- Twitter Card は `summary_large_image`
- og:title はタイプ名を含めた動的タイトルにする(例: 「ハラスメントタイプ診断 - あなたは『ハラスメント皇帝』」)

### 6-7. 共通ヘルパー関数

シェアテキストの組み立ては1箇所に集約する。

```typescript
// lib/harassment-type/share.ts
export function buildShareText(
  typeName: string,
  scores: { P: number; M: number; S: number; C: number },
  resultUrl: string
): string {
  return [
    `私のハラスメントタイプは「${typeName}」でした`,
    ``,
    `P:${scores.P}% / M:${scores.M}% / S:${scores.S}% / C:${scores.C}%`,
    ``,
    `#ハラスメントタイプ診断`,
    resultUrl,
  ].join('\n');
}
```

### 1. 無害な村人(typeId: `villager` / 属性: なし)

**カラー**: Main `#8A4F3A` / Pale `#F7EEE7`

**口癖**

- 「組織として問題だよ」
- 「再発防止策を出して」

- **「仕組みを変える」に焦点を絞ってみましょう**:同じ分析力でも、「あの人が悪い」で止まると断罪、「この仕組みが悪い」まで進むと改善になります。この一段を踏み越えるだけで、断罪する人から、仕組みを直せる人へと評価が変わっていきます。

### 14. 強欲の召喚士(typeId: `summoner` / 属性: PSC)

### 15. 執着の呪術師(typeId: `cursemaster` / 属性: MSC)

**カラー**: Main `#6D4A8C` / Pale `#F1ECF7`

**キャプション**
執着の呪術師は、親密さ、被害者意識、対応要求を絡めて、相手に罪悪感の呪いをかけるタイプです。直接命令するよりも、「前は違った」「大事にされていない」と感情をにじませることで、相手の行動を変えようとします。本人にとっては不安や寂しさの表現でも、相手にとっては、常に気持ちを証明し続けなければならない関係になります。

**口癖**
- 「前はもっと優しかったのに」
- 「私だけ雑に扱われてる気がする」
- 「……もういいです。気にしないでください」

**能力**
- **記憶の呪典**:以前の優しい言葉やLINEのやり取りを保存しておき、「あのときはこう言ってくれたのに」と現在の態度との落差を突きつける。
- **落ち込みによる拘束**:不満を直接命令にせず、悲しそうな反応で相手に追加対応を促す。

**アドバイス**
- **あなたの「人間関係に対する感受性の豊かさ」は、本当は稀少な感性です**:他人の言葉や態度を覚えていて、関係性を大切に育てたい気持ちが強いのは、希薄な人間関係の中ではむしろ貴重な資質。それを「過去で縛る」ではなく「思い出を一緒に積み重ねる」に向けてみてください。あなたは長く愛される人になれます。
- **気持ちを、呪いではなく言葉で伝えてみましょう**:寂しさや不満を、ため息や落ち込みで伝えるのをやめて、「私はこう感じている。だからこうしてほしい」と冷静に言語化する。これだけで、呪う人から、ちゃんと愛し愛される人へと関係性が変わっていきます。

### 16. ハラスメント皇帝(typeId: `emperor` / 属性: PMSC)

**カラー**: Main `#2B2A2E` / Pale `#F1F0F2`

**キャプション**
ハラスメント皇帝は、権力、正論、境界侵入、対応義務のすべてを統べるラスボス型です。相手の行動を詰め、人格を裁き、私生活に踏み込み、さらに説明・謝罪・特別対応まで求めます。本人の中では、すべてに正当な理由があるのかもしれません。しかし、その帝国では、相手の時間、感情、距離感、逃げ道までもが支配領域に含まれてしまいます。

**口癖**
- 「こっちは全部わかってるから」
- 「あなたのことを本当に心配してるから言ってるんだよ」
- 「最後にもう一度だけ確認させて」

**能力**
- **全方位包囲網**:仕事のミスを責めながら、態度・人格・私生活・対応姿勢まで一気に問題化する。
- **臣民総動員**:周囲の人、上司、担当者、関係者を巻き込み、相手が逃げにくい空気をつくる。

**アドバイス**
- **あなたには「視野の広さ」と「推進力」の両方があります**:物事を一面でしか見られない人が多い中、業務、ルール、関係性、対応など複数のレイヤーを同時に意識できるのは、強い才能です。それを支配ではなく統率に向けてみてください。あなたは組織を率いるリーダーとして本領を発揮できます。
- **「相手の自由」と「自分の権限」を、同じ重さで扱ってみましょう**:自分の権限が大きくなるほど、相手の自由を奪う影響力も大きくなる。これを自覚するのが第一歩です。命令する前に「これは命令か、お願いか、雑談か」を自分で分類してみてください。それができるようになったとき、皇帝から、慕われる王へと立場が変わっていきます。
